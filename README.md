## 📖 Overview

**AgriSense** turns a low-cost ESP32 sensor node into an end-to-end decision-support platform. Field sensors stream soil and climate readings over an **authenticated MQTT** channel to a cloud backend, which stores them in **MongoDB Atlas**, runs four **advanced deep-learning models**, and returns **crop, fertilizer, irrigation, and soil-fertility** recommendations — each paired with **explainable-AI (LIME)** insights and **bilingual (English + Nepali)** agronomic advice generated by Google Gemini, exportable as a professional **PDF report**.

The entire system is **containerized with Docker** and **continuously deployed to AWS EC2** through a GitHub Actions pipeline, served securely over **HTTPS** behind an nginx reverse proxy.

> 🌐 **Live application:** **https://smartagri.cloudcoesis.com**

## ✨ Highlights

- 🔌 **End-to-end IoT pipeline** — ESP32 → MQTT → FastAPI → MongoDB Atlas → dashboard, fully working in the cloud.
- 🔐 **Secure by design** — password-authenticated MQTT broker, HTTPS/TLS via Let's Encrypt, JWT + magic-link auth, RBAC history.
- 🧠 **Four advanced DL models** — a custom Sparse Weighted Fusion Transformer, an FT-Transformer, and two TabNet classifiers — 77–99% accuracy on Nepal-specific data.
- 🔍 **Explainable AI** — LIME feature-importance for fertilizer and soil predictions, so recommendations are transparent.
- 🗣️ **Bilingual AI advisor** — context-aware agronomic guidance in English and Nepali (Gemini 2.5 Flash) with a fully offline template fallback.
- 📄 **One-click PDF reports** — WeasyPrint with correct Devanagari shaping for farmer-ready documents.
- ☁️ **Production-grade DevOps** — Dockerized stack, Infrastructure-as-Code (Terraform), and push-to-deploy CI/CD on AWS.

## 🧠 Machine Learning — Phase 8–9 (Advanced DL, Nepal Profiles)

All models were retrained on **Nepal-specific crop profiles (18 crops)**, with **SMOTE** correcting class imbalance in the fertilizer dataset (5,780 → 21,060 samples).

| Model | Architecture | Task | Features | Classes | Accuracy |
|-------|-------------|------|----------|---------|----------|
| **SwiFT** | Sparse Weighted Fusion Transformer (custom PyTorch) | Crop recommendation | 13 | 18 crops | **77.1%** |
| **TTL** | FT-Transformer (Feature Tokenizer + Transformer) | Irrigation advice | 9 num + 2 cat | 5 levels | **98.5%** |
| **TabNet Soil** | pytorch-tabnet + LIME | Soil fertility | 5 | Low / Med / High | **98.7%** |
| **TabNet Fertilizer** | pytorch-tabnet + LIME | Fertilizer choice | 8 | 5 fertilizers | **98.3%** |

**Intelligence built in:**
- **Crop-aware irrigation** — the SwiFT crop prediction is chained into the TTL irrigation model in the `/full` and `/complete` flows.
- **Explainability (LIME)** — `/api/recommend/explain` returns per-feature importance for fertilizer and soil predictions.
- **FAO-56 ET₀** — Hargreaves–Samani reference evapotranspiration is used as a derived irrigation feature.
- **Resilient AI advice** — bilingual advice is generated in a **single concurrent Gemini call per section** (thread-offloaded so the API never blocks) and **falls back to offline templates** on rate-limits or outages.

## 🔐 Security

- **Authenticated MQTT** — `allow_anonymous false`; the broker password file is generated at container startup from environment secrets (no hashed credentials committed). The ESP32 and FastAPI both authenticate.
- **HTTPS everywhere** — nginx terminates TLS (Let's Encrypt) and redirects HTTP → HTTPS.
- **Layered app auth** — NextAuth email magic links + a NodeJS-issued `backend_token` JWT, enforced by an auth-guard component.
- **RBAC** — recommendation history is scoped per user; admins see all.
- **Secret hygiene** — all secrets live in gitignored `.env` files locally and in GitHub Actions secrets for deployment.

## ☁️ Deployment & DevOps

Push to `main` triggers the full pipeline:

```
git push origin main
        │
        ▼
 GitHub Actions (deploy.yml)
   1. discover EC2 by tag        4. rewrite MQTT host → "mosquitto"
   2. write .env from secrets     5. docker compose (prod) up --build -d
   3. rsync code to EC2           6. health check https://…/health
```

- **Containers (prod):** nginx · Next.js · Express · FastAPI · Mosquitto, on one Docker network.
- **Infrastructure as Code:** `infrastructure/` (Terraform modules for EC2, S3, IAM).
- **Required GitHub secrets:** `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `EC2_SSH_KEY`, `BACKEND_ENV`, `NODEJS_ENV`, `FRONTEND_ENV`.
  > `BACKEND_ENV` **must** contain `MQTT_USERNAME` and `MQTT_PASSWORD` — without them the authenticated broker cannot start.

## 🚀 Getting Started

### Prerequisites
Docker & Docker Compose · (or Python 3.13 + Node.js 20) · MongoDB Atlas account · OpenWeatherMap API key · *(optional)* Gemini API key.

### Run with Docker (recommended)
```bash
git clone https://github.com/Mukesh-Pant/smart-agriculture-iot.git
cd smart-agriculture-iot

# Create env files (see *.example in each service)
#   backend/.env · NodeJSbackend/.env · frontend/.env.local

# Build & launch the full stack
docker compose -f docker-compose.yml up --build -d
```
Frontend → `http://localhost:3000` · API docs → `http://localhost:8000/docs` · MQTT → `localhost:1883`

```bash
# Hot-reload development overlay
docker compose -f docker-compose.yml -f docker-compose.dev.yml up --build
```

### Run services individually
```bash
# FastAPI
cd backend && python -m venv venv && source venv/Scripts/activate
pip install -r requirements.txt
python ml/train_models.py            # train the 4 models
uvicorn app.main:app --reload        # :8000

# Express
cd NodeJSbackend && npm install && npm run dev   # :5000

# Next.js
cd frontend && npm install && npm run dev        # :3000
```

> 💡 **Windows + Docker Desktop:** if a native **Mosquitto Windows service** is installed it will seize IPv4 `:1883` ahead of the container. Disable it with `Stop-Service mosquitto; Set-Service mosquitto -StartupType Disabled`.

## 🔐 Authentication Flow

1. User enters email at `/login` → NextAuth sends a magic link (Gmail SMTP).
2. Link click → email verified.
3. First-time users complete `/onboarding` (name, device ID, district).
4. NodeJS issues a `backend_token` JWT cookie.
5. Dashboard is protected by `CheckAuth` (validates NextAuth session **and** JWT).
